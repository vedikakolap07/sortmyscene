# SortMyScene — Event Ticket Booking

A full-stack MERN application for browsing events, selecting seats, and booking tickets with a 10-minute reservation hold.

---

## Tech Stack

| Layer     | Technology                          |
|-----------|-------------------------------------|
| Backend   | Node.js, Express.js, MongoDB/Mongoose |
| Frontend  | React.js (CRA), React Router v6, Axios |
| Auth      | JWT (jsonwebtoken) + bcryptjs       |
| Validation| express-validator                   |

## How to Run

### 1. Backend

```bash
cd backend

# Copy and configure environment
cp .env.example .env
# Edit .env — set MONGODB_URI and JWT_SECRET

npm install
npm run seed      # Populate DB with 4 sample events + 80 seats each
npm start         # Runs on http://localhost:5000
# or: npm run dev  (requires nodemon: npm i -g nodemon)
```

### 2. Frontend

```bash
cd frontend

# Copy and configure environment
cp .env.example .env
# REACT_APP_API_URL=http://localhost:5000/api  (default)

npm install
npm start         # Runs on http://localhost:3000
```

Open [http://localhost:3000](http://localhost:3000) in your browser.

| Method | Route        | Auth | Description                              |
|--------|--------------|------|------------------------------------------|

| Method | Route          | Auth | Description                        |
|--------|----------------|------|------------------------------------|

**Body:** `{ reservationId }`

## Assumptions

1. **One active reservation per user per event** — reserving again cancels the previous hold and frees those seats.
2. **No payment gateway** — booking confirmation is the final step; `totalAmount` is stored but no payment is processed.
3. **Seat layout** — rows A–H × 10 columns per event (80 seats). Layout is generated by the seed script.
4. **User identity** — basic email/password auth. No email verification or OAuth for the scope of this task.
5. **No admin panel** — events and seats are managed via the seed script and directly in MongoDB.

## Design Decisions

### How double booking is prevented

The core mechanism is a **MongoDB transaction + atomic `updateMany` with a conditional filter**:

```js
// In POST /api/reserve (inside a session.withTransaction block):
const updateResult = await Seat.updateMany(
  { eventId, seatNumber: { $in: seatNumbers }, status: 'available' },
  { $set: { status: 'reserved' } },
  { session }
);

if (updateResult.modifiedCount !== seatNumbers.length) {
  // Fewer seats were updated than requested → at least one was already taken
  // Roll back the partial update within the same transaction
  await Seat.updateMany({ ... status: 'reserved' }, { $set: { status: 'available' } }, { session });
  throw conflict error;
}
```

Because the filter includes `status: 'available'`, two concurrent requests for the same seat will race, but only one can match and modify it. The loser sees `modifiedCount < requested` and the entire transaction is rolled back — **no partial bookings are ever committed**.

Booking confirmation (`POST /api/bookings`) performs a similar atomic check: it only updates seats with `status: 'reserved'` and validates that all were successfully marked `booked` before committing the Booking document.

### Expired reservations

- `Reservation.expiresAt` is indexed with `expireAfterSeconds: 0` (MongoDB TTL) so documents auto-delete after expiry.
- On `POST /api/bookings`, the server explicitly checks `reservation.expiresAt < new Date()` before proceeding and returns HTTP 410 if expired.
- The frontend `useCountdown` hook counts down client-side and calls `onExpired` to reset the UI, re-fetches seat data, and prompts the user to re-select.

### State management

- **AuthContext** (React Context + `useReducer`) handles user session globally. Token is stored in `localStorage` and re-validated on app load via `/api/auth/me`.
- **EventDetailPage** owns all local booking-flow state: `selectedSeats`, `step` (select → reserved → booked), `reservation`, `booking`.
- **useCountdown** is an isolated hook that takes `expiresAt` and returns a live `display` string + `expired` boolean with cleanup on unmount.
